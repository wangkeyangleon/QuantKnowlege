# PTOS 主升浪操作系统

> 业务设计初稿 · 优化修订版 V2.0  
> 2026年7月

## 一、设计目标

PTOS（Primary Trend Operating System）的目标，是从全市场中提前识别**可能由底部平台进入主升阶段**的股票，并在突破后的低风险确认点参与，而不是追逐已经完成大幅上涨的强势股。

系统完整闭环为：

> 全市场股票 → 每日扫描 → 涨停候选 → 主升评分 → 观察池 → 突破确认 → 20日回踩监测 → 二次启动 → 买入执行 → 趋势持有 → 分级退出

系统不以预测单日涨跌为目标，也不把“高分”直接等同于“可以买入”。候选质量、交易时机、市场环境和组合风险必须分别判断。

### 1.1 核心业务原则

1. **先门槛、后评分。** 首板、真实平台突破、有效板块趋势、数据完整性等关键条件不成立时，不允许用其他高分补偿。
2. **寻找刚启动，不追涨过头。** 涨幅、位置、连板、跳空和换手过热需要独立过滤。
3. **所有判断保持时间一致。** 计算当日信号时只能使用当日收盘前已经可获得的数据，禁止未来数据和不完整财务数据穿越。
4. **状态驱动。** 每只股票只能按合法路径流转；任何一次晋级、降级、失效和退出都应记录原因。
5. **信号必须可执行。** 买入信号需要经过涨跌停、停牌、流动性、T+1、滑点和仓位约束校验后，才能转换为订单与持仓。

## 二、整体业务架构

PTOS由五层组成：

1. **市场与数据层：** 日线、月线、复权因子、资金流、板块、基本面和交易日历。
2. **候选发现层：** 涨停扫描、首板识别、平台识别、突破确认、板块和基本面过滤。
3. **交易时机层：** 20个交易日回踩、缩量确认、二次启动识别。
4. **执行与组合层：** 信号校验、订单执行、仓位分配、行业集中度和风险预算。
5. **持有与退出层：** 趋势跟踪、减仓、清仓、归因和复盘。

推荐业务流转：

> LIMIT_UP_POOL → WATCH_LIST_POOL → MONITORING_POOL → BUY_SIGNAL → PORTFOLIO → EXIT

其中，评分决定候选的优先级，状态机决定候选是否具备进入下一阶段的资格。

## 三、全市场扫描与市场环境

### 3.1 基础股票池

每日收盘后扫描A股，基础过滤条件如下：

- 当日涨幅不低于9.8%；
- 排除ST、*ST和退市整理股票；
- 排除北交所股票；
- 排除停牌或无有效行情股票；
- 排除上市不足60个交易日的次新股；
- 排除成交额和成交量不足、无法满足实际下单规模的股票。

### 3.2 “已经涨太多”过滤

为避免系统偏离“发现未来主升浪”的目标，在进入观察池前增加过度上涨过滤。以下为首版参考阈值，后续由回测确定：

- 近20个交易日累计涨幅原则上不超过35%；
- 当前价相对120日最低价的涨幅原则上不超过80%；
- 过去60日涨停次数不超过1次，连续涨停直接淘汰；
- 大幅连续跳空、极端换手或明显放量滞涨时，不进入观察池；
- 若突破日收盘价距离平台顶部过远，视为追涨风险，不直接进入回踩监测。

过度上涨不是评分项，而是资格条件。任何一项触发时，应记录具体淘汰原因。

### 3.3 市场阶段过滤

市场环境分为 `RISK_ON`、`NEUTRAL`、`RISK_OFF` 三种状态，可综合宽基指数趋势、上涨家数、跌停家数、成交额和市场波动判断：

- `RISK_ON`：允许正常晋级和开仓；
- `NEUTRAL`：提高买入门槛并降低初始仓位；
- `RISK_OFF`：停止新增买入，只管理已有持仓和退出风险。

市场阶段不直接抬高个股评分，避免弱市中的个股高分掩盖系统性风险。

## 四、候选识别模块

### 4.1 涨停扫描器（Limit Up Scanner）

每日收盘扫描全部股票，将满足涨停阈值且通过基础过滤的股票写入涨停池。每条记录至少包含股票代码、名称、涨停日期、收盘价、涨幅、成交量、成交额和过滤结果。

### 4.2 首板识别器（First Limit Up Detector）

Mistery逻辑关注第一次有效启动，因此首板识别同时承担硬门槛和质量评分：

| 过去60日涨停情况 | 首板评分 | 业务处理 |
|---|---:|---|
| 第一次涨停 | 100 | 通过 |
| 第二次涨停 | 60 | 默认不通过，可作为研究样本 |
| 第三次涨停 | 20 | 淘汰 |
| 连续涨停或四次以上 | 0 | 淘汰 |

生产规则默认要求过去60日仅出现本次涨停。评分用于研究和排序，不得让连续板因其他模块高分而晋级。

### 4.3 月线平台识别器（Monthly Base Detector）

月线平台是系统的核心结构。使用评估日之前已经完整结束的月线数据，计算过去12个月最高价 `HH12`、最低价 `LL12` 和平台宽度：

> PlatformWidth = (HH12 - LL12) / LL12

基础要求为平台宽度小于25%。平台质量还应包括：

| 因素 | 参考条件 | 含义 |
|---|---|---|
| 平台持续时间 | 3至12个月 | 排除短期随机整理 |
| MA5月线斜率 | 绝对值小于3% | 中期趋势趋稳 |
| MA10月线斜率 | 绝对值小于2% | 长期趋势趋稳 |
| MA5与MA10距离 | 小于2% | 均线收敛 |
| 月ATR收缩 | ATR/Close下降 | 波动收敛 |
| 月成交量收缩 | 近6月均量低于近24月均量 | 筹码趋稳 |
| 平台上沿测试 | 至少2次接近上沿 | 平台顶部具有市场意义 |

平台反复跌破、下沿持续下移或仅由急跌后的弱反弹构成时，应降低平台评分或直接判定平台无效。

### 4.4 平台突破识别（Platform Breakout Recognizer）

突破锚点必须来自**评估日前的历史平台**。定义：

> PlatformTop = max(评估日前12个完整月的最高价)

不得把突破当日或当前未完成月份的最高价纳入 `PlatformTop`，否则会造成 `Close > HH` 在业务上无法成立，或形成前视偏差。

有效突破必须同时满足：

- 当日收盘价大于 `PlatformTop`；
- 当日成交量大于20日平均成交量的2倍；
- 收盘价不能距离平台顶部过远，默认不超过8%；
- 突破后1至3个交易日没有快速跌回平台内部；
- 记录 `is_breakout`、`platform_top_price`、`breakout_date` 和突破量比。

没有明确平台顶部和突破日期的股票，即使综合分很高，也不能进入监测池。

### 4.5 主力资金确认（Institutional Funds Confirmation）

资金流用于提高置信度，不作为单独买入依据。参考条件为连续5个交易日主力净流入为正，并同时检查累计净流入占成交额比例，避免小额流入被误判为强资金确认。数据缺失时应标记 `UNKNOWN`，不能按满分处理。

### 4.6 板块趋势过滤（Sector Trend Filter）

板块判断需要体现“主线效应”，不能只看个股所属行业的一组均线：

- 行业20日涨幅位于全行业前20%；或行业指数突破半年平台；
- 行业成交额和换手同步放大；
- 板块内有多只股票共同走强，不依赖单一股票拉动；
- 龙头股票已启动但板块整体尚未明显过热；
- 行业数据必须完整且为最新交易日数据。

板块趋势数据为空、过期或无法映射时，状态为 `UNKNOWN`，不得通过默认高分进入监测池。

### 4.7 基本面过滤（Fundamentals Filter）

基础质量要求：最近4个已披露季度营收同比增长大于15%、利润同比增长大于10%、ROE大于10%、资产负债率低于70%。金融、周期等行业应允许使用行业专属口径，避免统一阈值造成系统性偏差。

财务数据必须按实际披露日期生效，回测时不得使用当时尚未公开的报表。

## 五、评分与晋级规则

### 5.1 候选质量评分

建议将“候选质量”和“买入准备度”拆开。候选质量满分100分，首版权重如下：

| 模块 | 权重 |
|---|---:|
| 首板质量 | 10 |
| 月线平台质量 | 20 |
| 平台突破 | 20 |
| 突破成交量质量 | 15 |
| 板块趋势 | 15 |
| 基本面 | 10 |
| 主力资金 | 5 |
| 未过度上涨质量 | 5 |
| 合计 | 100 |

权重、阈值和缺失数据处理必须由配置管理并写入评分版本，确保回测和生产结果可复现。

### 5.2 进入观察池

进入观察池需要同时满足：基础股票池过滤通过、首板硬门槛通过、未过度上涨、候选质量评分不低于60分。观察池仅代表值得继续跟踪，不代表已经突破或可以买入。

### 5.3 进入监测池

股票进入回踩监测池必须同时满足：

- `overall_score >= 80`；
- `platform_breakout_score > 0`；
- `is_breakout = true`；
- `platform_top_price` 存在且大于0；
- `breakout_date` 存在且不晚于评估日；
- 板块趋势数据有效，`sector_trend_score > 0` 且板块为多头；
- 首板、过度上涨和市场阶段硬门槛仍然有效。

缺少关键数据时应判定为“不具备晋级资格”，而不是以0分、默认值或其他模块高分补偿。

## 六、回踩监控与二次启动

### 6.1 20个交易日监测生命周期

监测从 `breakout_date` 后第一个交易日开始，最长持续20个交易日。超过期限仍未形成有效回踩和二次启动时，状态转为 `EXPIRED`，退出监测池；如需再次进入，必须形成新的有效突破事件。

每个交易日同时监控三个回踩锚点：平台顶部、MA10和MA20。定义：

> Distance = abs(Low - AnchorPrice) / AnchorPrice

当距离不超过3%时视为触达锚点。优先级为平台顶部，其次MA10、MA20，并记录实际触发的锚点类型。

### 6.2 回踩有效性

有效回踩需要满足：

- 价格接近至少一个有效锚点；
- 收盘价没有有效跌破平台顶部，默认失效线为 `PlatformTop × 0.97`；
- 回踩过程没有连续放量长阴或明显派发；
- 板块趋势没有转为空头，市场不处于 `RISK_OFF`；
- 回踩期间最低价、最低价日期、距离和状态每日更新。

一旦有效跌破平台或出现结构性破坏，立即转为 `INVALIDATED` 并停止等待买点。

### 6.3 缩量确认

回踩期间成交量低于20日均量是最低要求，同时建议计算最近3日平均量与突破日成交量之比。成交量逐步收缩、价格波动收敛、下影承接增强时，提高买入准备度；仅单日缩量不能单独确认。

### 6.4 二次启动识别（Second Leg Up Detection）

二次启动是最终的交易触发点，必须同时满足：

- 当日涨幅大于3%；
- 当日成交量大于20日均量的1.5倍；
- 收盘价重新站上MA5和MA10；
- 收盘价接近或重新突破回踩后的局部高点；
- 平台结构仍有效，板块和市场环境没有恶化；
- 当日股票可正常交易且具备足够流动性。

满足条件后生成 `BUY_SIGNAL`，至少记录信号日期、信号价格、触发原因、平台顶部、突破日期、置信度、评分版本和有效期限。若次日大幅高开导致预期盈亏比明显恶化，订单应取消或降级，不能无条件追价。

## 七、状态机

推荐状态如下：

> WATCH → LIMIT_UP → WATCH_LIST → BREAKOUT_CONFIRMED → MONITORING → PULLBACK → READY → BUY_SIGNAL → HOLD → REDUCE / ADD → EXIT

补充终止状态：`REJECTED`、`INVALIDATED`、`EXPIRED`、`ORDER_FAILED`。

关键流转规则：

| 当前状态 | 进入下一状态的条件 | 失败处理 |
|---|---|---|
| LIMIT_UP | 首板、基础过滤和未过热条件通过 | REJECTED |
| WATCH_LIST | 总分达标且真实突破条件成立 | 保持观察或REJECTED |
| MONITORING | 20日内完成有效回踩 | INVALIDATED或EXPIRED |
| PULLBACK | 缩量和价格结构确认 | 继续等待或INVALIDATED |
| READY | 二次启动全部条件成立 | 保持READY或EXPIRED |
| BUY_SIGNAL | 订单校验并成交 | ORDER_FAILED或重新评估 |
| HOLD | 趋势保持 | REDUCE、ADD或EXIT |

状态更新应保存原状态、新状态、事件时间、规则版本和原因码，保证可审计和可复盘。

## 八、买入执行与组合风控

### 8.1 信号到持仓

系统必须实现 `BUY_SIGNAL → ORDER → FILL → PORTFOLIO` 的闭环。订单执行前检查停牌、涨停无法成交、最小交易单位、可用资金、滑点、手续费、T+1和重复订单。

未成交信号不得直接写成持仓；部分成交应按实际成交数量和均价建仓。

### 8.2 初始仓位与加仓

以下为首版风险参数，必须配置化并通过回测校准：

- 单只股票最大仓位不超过组合净值的10%；
- 首次买入使用计划仓位的50%，即默认初始仓位不超过5%；
- 同一行业合计仓位不超过30%；
- 同时持仓原则上不超过8只；
- 只在已有浮盈、趋势再次确认且组合风险允许时加仓，禁止亏损摊平；
- `RISK_OFF` 状态下总仓位降至30%以下并停止新开仓。

若组合从净值高点回撤达到预设阈值，或连续亏损达到限制，应自动降低风险预算并暂停新信号执行。

## 九、趋势持有与卖出策略

卖出不采用固定止盈，而采用结构破坏和趋势弱化的分级处理。执行优先级为：全部卖出 > 减仓 > 持有，避免同一交易日重复触发多次减仓。

| 条件 | 动作 | 说明 |
|---|---:|---|
| 收盘有效跌破平台失效线 | 全部卖出 | 核心结构被破坏 |
| 月线收盘跌破MA5 | 全部卖出 | 中期趋势结束 |
| 日线持续跌破MA60 | 全部卖出 | 趋势安全退出 |
| 日线跌破MA20 | 减仓30% | 趋势转弱预警 |
| MACD死叉且动能同步转弱 | 减仓50% | 避免单一指标误触发 |
| 板块跌出强势区且个股相对强度持续下降 | 降低仓位或退出 | 个股主升环境消失 |
| 放量冲高回落、异常巨量滞涨 | 收紧止盈或减仓 | 可能出现派发 |

卖出规则应明确“收盘确认”还是“盘中触发”、连续确认天数、同日信号合并方式和T+1限制。平台失效、月线MA5和MA60属于清仓规则；MA20和MACD属于风险降低规则。

## 十、数据与数据库设计

行情、板块和财务历史数据与业务状态分离。行情数据按股票、频率和日期存储；数据库只保存池子、评分、状态、信号、订单和持仓。

### 10.1 核心业务表

| 表 | 作用 | 关键字段 |
|---|---|---|
| limit_up_pool | 当日涨停候选 | code、date、price、pct_change、filter_result |
| watch_list_pool | 候选质量与平台信息 | scores、HH、LL、is_breakout、platform_top_price、breakout_date |
| monitoring_pool | 20日回踩生命周期 | start_date、expiry_date、anchor_type、distance、lowest_price、status |
| buy_signal | 买入事件 | signal_date、signal_price、confidence、reason、expires_at |
| portfolio | 实际持仓 | quantity、avg_price、position_weight、platform_top、entry_signal |
| stock_stage | 当前状态快照 | stage、stage_date、reason_code、rule_version |
| stage_event | 状态变更日志 | from_stage、to_stage、event_time、reason、payload |
| trend_score | 每日评分明细 | factor_scores、total_score、data_date、score_version |
| orders / fills | 委托与成交 | order_status、quantity、price、fees、slippage |

所有业务表使用统一股票代码、交易日和规则版本。关键价格字段必须校验非空且大于0；日期必须不晚于当前评估日。

## 十一、回测、监控与验收

### 11.1 回测约束

- 使用点时数据，财务指标按披露日生效；
- 月线平台只使用已经完整结束的月份；
- 处理复权、分红、拆股、停牌、涨跌停和退市股票；
- 纳入手续费、印花税、滑点和无法成交情形；
- 使用滚动训练/验证区间，避免只对单一牛市调参；
- 保留被淘汰股票，避免幸存者偏差。

### 11.2 业务监控指标

系统至少持续统计：

- 涨停池到观察池、监测池、买入信号和成交的阶段转化率；
- 突破后20日内的最大有利波动（MFE）和最大不利波动（MAE）；
- 信号胜率、盈亏比、期望收益、持有周期和换手率；
- 最大回撤、行业暴露、单票集中度和连续亏损次数；
- 信号产生到下单、成交和状态更新的延迟；
- 因缺失数据、无锚点、板块无效、过度上涨和订单失败被拒绝的数量。

评估重点不是单纯胜率，而是扣除成本后的期望收益、回撤和在不同市场阶段的稳定性。

## 十二、与 AiQuant V9 的集成

PTOS作为核心Alpha来源，与其他模块协同：

> Market Phase + Sector Rotation + News/Risk Events → PTOS候选与买点 → Risk Engine → Portfolio Engine → Auto Trading

- Market Phase决定是否允许开仓及风险预算；
- Sector Rotation提供行业排名、扩散度和主线强度；
- News/Risk Events用于识别停牌、监管、减持、业绩和重大事件风险；
- Risk Engine校验单票、行业、组合回撤和流动性限制；
- Portfolio Engine负责排序、资金分配、订单冲突和持仓再平衡；
- Auto Trading回传委托、成交和失败原因，形成完整闭环。

## 十三、首版实施优先级

### P0：保证业务链路正确

1. 修正平台顶部口径，排除突破日和未完成月份；
2. 落实监测池晋级硬门槛，禁止无突破锚点和板块空数据高分晋级；
3. 完成20个交易日监测、失效和过期逻辑；
4. 二次启动补充当日涨幅大于3%的条件；
5. 打通买入信号、订单、成交和持仓；
6. 所有评分与阈值配置化并保存版本。

### P1：提升业务稳定性

1. 增加过度上涨过滤和首板硬门槛；
2. 增加市场阶段和板块主线强度；
3. 增加组合仓位、行业集中度和回撤控制；
4. 完善平台破位、MA20、MA60、月线MA5和板块退潮退出规则；
5. 建立状态事件日志和业务监控指标。

### P2：形成可验证的策略产品

1. 建立点时回测、交易成本和不可成交模拟；
2. 按市场阶段和行业进行分层评估；
3. 使用滚动样本校准阈值与权重；
4. 建立信号归因、每日复盘和参数漂移监控；
5. 在仿真盘稳定运行后，再逐步开放自动交易权限。

## 十四、参数管理说明

本文中的9.8%、25%、35%、80%、3%、2倍、1.5倍、20个交易日等均为首版业务参数，不应写死在代码中。每项参数需要具备配置名称、默认值、生效日期、版本号和修改记录，并通过历史回测、仿真盘和小规模实盘逐级验证。

PTOS的最终判断顺序应保持稳定：

> 数据有效 → 硬门槛通过 → 候选评分达标 → 突破锚点成立 → 回踩缩量确认 → 二次启动 → 执行与组合风控 → 趋势持有与退出

只有当这条业务链完整闭环时，系统才真正符合“提前发现主升浪，并在可控风险下参与”的设计目标。
